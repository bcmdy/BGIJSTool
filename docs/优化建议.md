# BGIJSTool 优化建议

> 本文档以“使用者 + 维护者”双重视角，审查当前代码后整理出可优化项。
> 每条标注**优先级**（🔴 高 / 🟡 中 / 🟢 低）、问题、影响和建议方案，并给出涉及的文件位置。
> 评审日期：2026-06-16。更新日期：2026-06-20（高/中/低优先级项均已落地，详见文末“完成情况”）。

## 一、用户体验（UX）

### 1.1 🔴 执行期间界面卡死（无异步）
- **位置**：[MainWindow.xaml.cs:186-214](../MainWindow.xaml.cs#L186-L214)、[MainWindow.xaml.cs:216-242](../MainWindow.xaml.cs#L216-L242)
- **问题**：`ExecuteBtn_Click` / `ExecuteRestoreBtn_Click` 直接在 UI 线程上同步执行所有文件 IO（遍历目录、复制文件、压缩/解压 zip）。虽然调用了 `SetBusy(true, "执行中...")`，但因为后续操作没有让出 UI 线程，状态文字和按钮禁用状态在操作完成前根本不会重绘，界面表现为“假死”。
- **影响**：备份/还原文件较多或 zip 较大时，窗口无响应、无法看到进度，用户容易误以为程序崩溃而强制关闭。
- **建议**：
  - 将耗时操作放到后台线程：`await Task.Run(() => operationService.ExecuteModule(module))`，事件处理器改为 `async void`。
  - `Logger` 已用 `Dispatcher.Invoke` 回写 UI，后台线程写日志是安全的，迁移成本低。
  - 配合进度反馈（见 1.2）。

### 1.2 🟡 缺少进度反馈
- **问题**：批量操作只有开始/结束日志，没有“正在处理第 N/总数”的进度。
- **建议**：`ILogger` 增加进度回调，或在状态栏显示 `已处理 X/Y`。结合 1.1 的后台线程即可实时刷新。

### 1.3 🔴 执行删除（del）前没有二次确认
- **位置**：[MainWindow.xaml.cs:186](../MainWindow.xaml.cs#L186)（执行）对比 [MainWindow.xaml.cs:252-258](../MainWindow.xaml.cs#L252-L258)（删除备份已有确认）
- **问题**：删除**备份 zip** 时弹了确认框，但“执行模块”里包含 `del` 操作会**永久删除用户的真实脚本文件**，却没有任何确认。虽然通常先 `bak` 再 `del`，但若模块配置只有 `del`（README 路径写法允许）、或备份步骤未命中文件，用户数据会直接丢失。
- **建议**：执行包含 `del` 的模块前弹确认框，列出将删除的文件数/是否已生成备份。

### 1.4 🟡 无法在程序内预览/试运行（dry-run）
- **问题**：用户只能凭 `config.json` 想象会发生什么。选中模块虽在日志里列出步骤路径（[MainWindow.xaml.cs:154-168](../MainWindow.xaml.cs#L154-L168)），但展示的是配置里的原始通配符，不是**实际会命中的文件**。
- **建议**：增加“预览/试运行”按钮，调用 `ResolveBgiPaths` 展开后显示真实文件清单与操作类型，不做实际改动。

### 1.5 🟡 模块/配置只能手动改 JSON
- **问题**：所有模块都要按 `SPEC.md` 手写 `config.json`，程序内只能选 BetterGI 路径，无法增删模块。门槛偏高，易写错。
- **建议**：（中期）提供简单的模块编辑界面，或至少“打开 config.json”和“校验配置”按钮（当前校验仅在加载时进入日志）。

## 二、正确性与安全

### 2.1 🔴 还原清单的 `del` 路径未做路径穿越校验
- **位置**：写入清单 [FileManager.cs:202-227](../FileManager.cs#L202-L227)，还原时删除 [FileManager.cs:319-325](../FileManager.cs#L319-L325)
- **问题**：备份时把 copy zip 内的条目名原样记入清单的 `del` 列表（`copyDeletedPaths`，仅替换了斜杠，未过滤 `..`）。还原时直接 `DeleteFile(p)` → `GetFullPath(p)` 拼接路径并删除（[FileManager.cs:620-628](../FileManager.cs#L620-L628)）。若 copy zip 内含 `..\..\x` 之类条目名，还原时会**删除 `JsScript` 目录之外的文件**。
- **对比**：解压 copy 时有 `IsSafePath` 校验（[FileManager.cs:451](../FileManager.cs#L451)），但记录到清单的删除路径走的是另一条路径，绕过了校验。
- **建议**：写入清单前对 `copyDeletedPaths` 同样调用 `IsSafePath`/拒绝含 `..` 的条目；还原删除前再校验一次（防御纵深）。

### 2.2 🟡 `IsSafePath` 前缀比较存在边界缺陷
- **位置**：[FileManager.cs:481-502](../FileManager.cs#L481-L502)
- **问题**：`fullPath.StartsWith(fullDestDir, OrdinalIgnoreCase)` 未确保 `fullDestDir` 以目录分隔符结尾。理论上 `C:\dest` 会匹配到 `C:\dest_evil`。当前因前面已拒绝 `..`，实际影响有限，但属于脆弱写法。
- **建议**：比较前把 `fullDestDir` 规范化为以 `Path.DirectorySeparatorChar` 结尾，或用 `Path.GetRelativePath` 判断是否以 `..` 开头。

### 2.3 🟡 GBK 编码识别依赖硬编码中文词表，无法通用
- **位置**：`IsPlausiblePath` 的 `chineseMarkers`（[FileManager.cs:563-584](../FileManager.cs#L563-L584)）、`ContainsMojibake`（[FileManager.cs:540-557](../FileManager.cs#L540-L557)）
- **问题**：编码判定里写死了“稻妻/须弥/狗粮/茶叶/智障”等游戏术语。换一批脚本、其他用户的 zip、非中文路径就可能误判，把正确文件名当乱码“修正”坏，或反之。
- **建议**：改用更通用的启发式——读取 zip 条目的**通用位标志（general purpose bit 11）**判断是否声明了 UTF-8；否则按字节用 GBK/GB18030 解码并用“是否落在合法 Unicode 区间/是否产生替换字符”判断，去掉领域词表。

### 2.4 🟢 跨午夜日志仍写入旧日期文件
- **位置**：[Logger.cs:18](../Logger.cs#L18)
- **问题**：日志文件名在构造时固定为当天日期。程序长时间运行跨过 0 点后，仍写入前一天的 `.log`。
- **建议**：写入时按 `DateTime.Now` 计算当天文件路径，而非构造时缓存。

## 三、代码结构与可维护性

### 3.1 🔴 `FileManager` 内存在与 `ScriptOperationService` 重复的编排逻辑（疑似死代码）
- **位置**：`FileManager.ExecuteSteps` / `ExecuteStep` / `ExecuteDel` / `ExecuteCopyStep` / `BackupFile`（[FileManager.cs:119-160](../FileManager.cs#L119-L160)、[FileManager.cs:604-618](../FileManager.cs#L604-L618)）
- **问题**：`FileManager` 既是“文件操作原语”又自带一套步骤编排（`ExecuteSteps` 复刻了 `ScriptOperationService.ExecuteModule` 的 bak/del/copy 收集与执行流程，且 zip 命名逻辑还不一样）。而实际 UI 走的是 `ScriptOperationService`。这套并行实现是冗余的、容易与真正用到的逻辑产生不一致，增加维护负担。
- **建议**：明确职责——`FileManager` 只保留文件原子操作（拷贝/删除/解压/读备份信息），删除 `ExecuteSteps`/`ExecuteStep`/`ExecuteDel`/`ExecuteCopyStep`/`BackupFile` 等未被使用的编排方法。删除前用 `Grep` 确认无引用。

### 3.2 🟡 `ScriptOperationService.ExecuteModule` 与 `FileManager` 职责边界模糊
- **位置**：[ScriptOperationService.cs:16-76](../ScriptOperationService.cs#L16-L76)
- **问题**：`ExecuteModule` 自己做了路径解析与备份聚合（`ResolveBgiPaths` + `backupPaths`），又把 copy 拆成单路径 Step 逐个调 `FileManager.ExecuteCopy`（[ScriptOperationService.cs:101-105](../ScriptOperationService.cs#L101-L105)），而 `FileManager.ExecuteCopy` 只取 `paths[0]`，调用方式绕。
- **建议**：让 `FileManager.ExecuteCopy` 直接接受路径列表并内部遍历；编排层只负责“按 op 分发”，不重复解析。

### 3.3 🟡 `ILogger` 接口耦合 WPF
- **位置**：[ILogger.cs:1](../ILogger.cs#L1) `using System.Windows.Controls;`
- **问题**：接口本身不暴露任何 WPF 类型，却 `using System.Windows.Controls`，是无意义的耦合。这也间接造成 `ScriptOperationService` 难以在测试中注入假 logger（虽然接口可实现，但容易让人误以为依赖 UI）。
- **建议**：删除该 using；把 `ILogger` 移到不依赖 UI 的位置，便于服务层与测试复用。

### 3.4 🟡 大量空 `catch {}` 吞异常
- **位置**：[FileManager.cs:139](../FileManager.cs#L139)、[219](../FileManager.cs#L219)、[238](../FileManager.cs#L238)、[259](../FileManager.cs#L259)、[343](../FileManager.cs#L343)、[362](../FileManager.cs#L362)、[520](../FileManager.cs#L520)、[531](../FileManager.cs#L531)
- **问题**：临时目录清理、zip 读取等失败被静默吞掉。最直接的后果：`_tmp_bak_*` / `_restore_tmp_*` 临时目录在清理失败时会**永久残留在 `backup/` 目录**，越积越多且会被 `GetBackupInfos` 扫到（虽然只匹配 `*.zip`，但目录垃圾仍在）。
- **建议**：清理失败至少 `logger.LogWarning`；zip 读取失败要给出可诊断的提示，而非完全静默。

### 3.5 🟢 硬编码中文字符串、颜色，无本地化
- **位置**：遍布 `MainWindow.xaml.cs`、`Logger.cs`（颜色映射 [Logger.cs:26-33](../Logger.cs#L26-L33)）
- **建议**：小工具可暂不做 i18n，但可把日志级别→颜色、状态文案集中到常量/资源，减少散落的魔法字符串。

## 四、性能

### 4.1 🟡 日志每行都新建 `BrushConverter` 并解析颜色
- **位置**：[Logger.cs:54-67](../Logger.cs#L54-L67)
- **问题**：每写一条日志都 `new BrushConverter().ConvertFromString(color)`。批量操作动辄上百行日志，重复解析+创建画刷开销可观，且未 `Freeze`。
- **建议**：预先构造并 `Freeze` 4 种级别画刷缓存（信息/成功/警告/错误），按类型取用。

### 4.2 🟡 每次刷新备份列表都重新打开所有 zip
- **位置**：`GetBackupInfos` → `ReadBackupInfo`（[FileManager.cs:53-102](../FileManager.cs#L53-L102)），在加载、执行、还原、删除后均调用 `PopulateRestoreCombo`（[MainWindow.xaml.cs:280](../MainWindow.xaml.cs#L280)）
- **问题**：每次都逐个打开 zip 读取 `_restore_manifest.json`。备份多时每次刷新都有明显 IO。
- **建议**：按 `FullPath + LastWriteTime` 缓存解析结果，文件未变则复用。

### 4.3 🟢 `FileManager` 频繁重新实例化
- **位置**：`CreateOperationService`、`PopulateRestoreCombo`（[MainWindow.xaml.cs:283](../MainWindow.xaml.cs#L283)、[368](../MainWindow.xaml.cs#L368)）
- **问题**：每次操作都 new 一个 `FileManager`，构造函数还会 `Directory.CreateDirectory` 两次。影响很小，但可在路径变更时才重建。

## 五、测试

### 5.1 🟡 关键安全/编码逻辑缺测试
- **现状**：现有测试只覆盖 `SanitizeFileName`、`GetBackupInfos`、`ConfigService` 校验（[FileManagerTests.cs](../BGIJSTool.Tests/FileManagerTests.cs)、[ConfigServiceTests.cs](../BGIJSTool.Tests/ConfigServiceTests.cs)）。
- **缺口**：
  - `IsSafePath` / 路径穿越（含 2.1、2.2 的回归用例）。
  - GBK/UTF-8 编码识别（2.3）。
  - `ScriptOperationService.ExecuteModule` 的 bak→del→copy→restore 端到端流程。
  - `ResolveBgiPaths` 的通配符与目录递归展开。
- **建议**：补充上述单元测试，尤其把 2.1 的传越场景固化为回归用例。`IsSafePath` 等可改为 `internal` + `InternalsVisibleTo` 以便测试。

## 六、CI / 工程

### 6.1 🟢 Release 工作流缺独立 CI（PR 校验）
- **位置**：[.github/workflows/release.yml](../.github/workflows/release.yml)
- **问题**：仅在 `release`/手动触发时构建+测试，没有 push/PR 触发的持续集成，问题要到发版才暴露。
- **建议**：增加 `on: [push, pull_request]` 的轻量 CI（build + test），与发布流程分离。

### 6.2 🟢 `.zip` 二进制制品入库
- **位置**：仓库根 `copy/*.zip`
- **问题**：示例/实际 copy 包以二进制形式进入 git 历史，仓库会随版本变大。
- **建议**：评估是否用 Git LFS，或仅在 release 制品中附带、从源码树移除。
- **评估结论（2026-06-20）：维持现状，不改动。** 三个 zip 合计仅 32K、仅提交过一次（无反复变更历史），且是程序运行所需的功能性资源（copy 步骤解压、release 工作流随包发布、`config.json` 按名引用）。在此体量下引入 Git LFS 或移出源码树只会增加协作/发布复杂度且可能破坏本地运行与发版，无实际收益。
